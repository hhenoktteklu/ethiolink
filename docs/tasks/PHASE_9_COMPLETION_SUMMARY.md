# Phase 9 — Completion Summary

End of Phase 9. The platform now has every Phase-9 engineering surface in place: a real SMS gateway path, a Telegram bot path, customer + business-owner Flutter apps, full English + Amharic localization (mobile UI + notification templates), customer-managed-key infrastructure with a documented migration runbook, marketplace full-text search (backend + mobile), and paid featuring end-to-end (subscriptions, sweep Lambda, owner mobile UI, admin SPA panel, operator runbook). Every track has a documented operator playbook for the rollout step it depends on; the open work is operator-led, not code-led.

Authoritative scope and per-track sub-checklists live in [`PHASE_9_POST_MVP.md`](./PHASE_9_POST_MVP.md). This document is the at-a-glance status read on 2026-05-16.

## Goal recap

Phase 9 was the "post-Phase-8 push toward launch" — the work between *"the platform runs"* (Phase 8 close) and *"customers can find a salon on their phone, book a haircut, and get a reminder in their language"*. The phase entry scoping (see [`PHASE_9_POST_MVP.md`](./PHASE_9_POST_MVP.md) § "Goals") expressed it as three rough buckets:

- **Reach paying customers.** Real SMS notifications, real Telegram fallback, customer + owner mobile apps, end-to-end booking on a phone.
- **Make the platform usable for an Amharic-speaking audience.** Locale-aware notification templates, ARB-bundled mobile UI, locale picker, server-side preference persistence.
- **Lay the groundwork for sustainable growth.** Customer-managed encryption (KMS), marketplace search improvements, paid featuring as the first revenue-bearing feature.

All three buckets shipped on the engineering side. The remaining gates (real provider rollouts, native-speaker translation review, real-device smokes, k6 captures, DR tabletop) are operator-led and the four runbooks shipped this phase document each one step-by-step.

## Current product status

End-of-Phase-9 capability picture:

- **Customers can find and book a salon end-to-end from a phone.** The Flutter app's customer surface covers every action in `MVP_SCOPE.md`: browse → category → business → service → date → slot → confirm → history → cancel / review. Search is live and Amharic-aware.
- **Business owners can run their listing end-to-end from the same phone.** Create business → submit for review → manage services / staff / availability → process bookings → edit profile → promote (paid featuring). No admin-SPA dependency for the owner workflow.
- **Notifications reach customers and owners in their preferred language** through the SMS + Telegram dual-path dispatcher with `TELEGRAM > SMS > MOCK` per-recipient priority. Notification logs record every attempt with `status` + `provider_ref` for reconciliation.
- **Admins manage the marketplace and revenue from the admin SPA.** Categories CRUD, business approval / rejection / suspension / featuring, user suspension / restore, cross-business appointments listing, notification logs, paid-featuring history + comp + cancel. Manual `featured_until` write remains as the operator escape hatch.
- **Marketplace growth surfaces are live.** Full-text search across English + Amharic descriptions, ranked relevance with featured-first as the default sort, paid featuring as the first revenue-bearing feature.

## Deferred post-Phase-9 items

Tracked so we don't lose them. Each is an independent workstream; none are gating each other.

- **Real Telebirr / Chapa gateway.** The `PaymentGateway` abstraction (with discriminated `purpose: 'APPOINTMENT' | 'FEATURING'` input) and `MockOnlineGateway` test seam are in place. Adding the real Ethiopian payment-rail provider retires `ONLINE_PAYMENTS_UNAVAILABLE` as a placeholder error and unblocks real revenue capture on paid featuring + the future online-booking path. Estimate: 1–2 weeks for gateway implementation + Secrets Manager rotation playbook + handler integration tests + a paired dev-smoke.
- **Service-level search.** Today's search ranks against business `name` + `description`. Expanding the index to also match against `services.name_en` + `services.name_am` lets a customer search "haircut" and surface every business that lists a haircut service — a meaningful precision win at modest schema cost.
- **Search analytics dashboard.** No telemetry today on what customers search for, which results they tap, or which sort modes they pick. An admin-SPA tab over a new `search_queries` table + a `GET /v1/admin/search/analytics` summary endpoint surfaces query → tap → book funnels and informs the next-round ranking work.
- **Receipts / invoices.** Owners can't currently print or email a receipt after a successful featuring purchase (or, eventually, an online booking). A `GET /v1/businesses/{id}/featuring/subscriptions/{subId}/receipt` PDF endpoint + an "Email receipt" button on the owner mobile featured-header closes the loop. Pairs naturally with the real-payment-provider work.
- **Push notifications (FCM + APNs).** Out of MVP per `MVP_SCOPE.md`. Currently SMS + Telegram covers the real-time delivery story; native push is the post-launch upgrade for users who prefer it. Telegram link rates among Ethiopian customers should inform the priority.
- **Richer booking rows with display names.** Today's appointment rows show `customerId` / `serviceId` / `staffId` UUIDs in the owner inbox. Joining + projecting the display names (customer email or display-name, service `name.en`, staff `display_name`) is a pure-presentation win. Pairs with a future `AppointmentViewWithJoins` projection.
- **Business analytics.** Read-only dashboard for owners showing booking volume, revenue split by payment method, top services by booked-volume, repeat-customer rate. Sits in the owner mobile app as a new dashboard card and on the admin SPA as a per-business detail tab. Requires a small analytics roll-up (Postgres view or a daily aggregator) — not a separate warehouse.

## Final recommendation

Two paths, pick by stakeholder priority.

- **If launch is the priority** — execute the operator-led launch checklist above. Twelve items; expected wall-clock of one to two weeks for an unblocked operator with provider relationships in place. Engineering's job in this path is on-call for any surprises the operator surfaces; no new code lands unless a regression appears. The runbooks shipped this phase are designed to be standalone — every line is operator-actionable without engineering hand-holding.

- **If code is the priority** — start Phase 10 with the **real payment provider integration** (`TelebirrGateway` or `ChapaGateway` replacing `MockOnlineGateway`). It's the highest-impact deferred item: closes the largest known gap in the paid-featuring surface, retires the `ONLINE_PAYMENTS_UNAVAILABLE` placeholder, and unblocks real revenue capture. The `PaymentGateway` abstraction is ready; what's missing is the concrete vendor integration plus the rotation / secret-management story. Estimate: 1–2 weeks. Phase 10's broader scope can then assemble around it (online booking payments, the receipts surface, search analytics, business analytics).

Both paths converge on the same end-state — a marketplace that's live, monitored, and capturing revenue. The split is purely about which surface ships next.
